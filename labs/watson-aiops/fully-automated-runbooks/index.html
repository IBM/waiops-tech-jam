12. Fully Automated Runbooks

12.1: Overview​

IBM Runbook Automation can automate procedures that can help IT staff solve everyday operational problems and do not require human interaction, thereby increasing the efficiency of IT operational processes.

Runbooks start as documented procedures on a piece of paper that can become fully automated procedures.

This module focuses on the creation of a fully-automated runbook and updates runbook results in the event journal.

Scenario: There is a apache2 server running on HOST. The apache2 process is monitored and configured to generate an alert when the apache2 API service goes UP/DOWN.

Objective: We have a manual runbook available (created in previous lab) for this problem and have a good success rate based on feedback. As a next step to mature this runbook and achieve full automation, we want to execute this runbook automatically and update results in the event journal. So the problem can be fixed without operator intervention.

By the end of this module, you will have a fully automated runbook ready in the Watson AIOPs Event Manager environment.

12.2: Enable ObjectServer REST Interface​

In order to update event journal runbook sends update via REST API to Objectserver. ObjectServer REST interface is disabled out of the box for Objectserver. So we need to first enable Objectserver REST interface using below steps:

a) Find ConfigMap for the primary ObjectServer

oc get configmap| grep primary
Example Output:
evtmanager-objserv-agg-primary-config 2 14d

b) Edit primary ObjectServer Configmap to enable REST interface in properties.


oc edit configmap evtmanager-objserv-agg-primary-config

Add the following to the ConfigMap:

apiVersion: v1
data:
  agg-p-props-append: |
    NRestOS.Enable: TRUE
    NHttpd.EnableHTTP: TRUE
    NHttpd.ListeningPort: 8080
  agg-p-sql-extensions: ""

c) Restart primary ObjectServer pod

oc delete pod evtmanager-ncoprimary-0

d) Expose Listening port for REST API

oc expose po evtmanager-ncoprimary-0 --port=8080 --type=NodePort --name=objserver-http-client-external-portforward

Check exposed port and make note of the port assigned.

oc get svc | grep objserver-http-client-external-portforward

e) Retrieve OMNIBUS root password :

oc get secret evtmanager-omni-secret -o go-template='{{range $k,$v := .data}}{{printf "%s: " $k}}{{if not $v}}{{$v}}{{else}}{{$v | base64decode}}{{end}}{{"\n"}}{{end}}'

f) You need to delete 1 network policy in the cluster in order to successfully run next curl command that connects to ObjectServer REST API

oc delete networkpolicy evtmanager-ibm-netcool-prod-network-policy

g) Test REST interface, OMNIbus password and port from the step (e) and step (d) are used to test the REST interface using curl.

Cluster name is : <Your cluster ingressdomain> like "tech-jam-xxx.containers.appdomain.cloud".

curl -k -u root:<OMNIBUS password from step e> -X POST -d "{ \"sqlcmd\": \"select count(*) from alerts.status\" }" -H 'Accept: application/json' -H 'Content-Type: application/json' http://<ClusterHostname>:<NodePort from step d>/objectserver/restapi/sql/factory
Example Output:
{ "rowset": { "osname": "AGG_P", "affectedRows": 1, "coldesc": [{
"name": "COUNT( * )", "type": "integer", "size": 4 }], "rows": [{ "COUNT( * )":
110 }] } }

12.3: Create automation to update the journal​

Create an automation "Update Journal" with two additional parameters "alertid" and "message".

Navigate to: Automations → Runbooks → Automations → Create Automation

Script:

curl -k -u root:<OMNIBUS password from step e> -X POST -d "{ \"sqlcmd\": \"exec jinsert($alertid,0, getdate, '$message')\" }" -H 'Accept: application/json' -H 'Content-Type: application/json' "http://<ClusterHostname>:<NodePort from step d>/objectserver/restapi/sql/factory"

Save automation "Update Journal".

12.4: Update Runbook - add a step to update the journal​

Navigate to: Automations → Runbooks → Library → Edit runbook "Restart API Service"

Add Automated Step -> Select "Update Journal"

Add parameter mapping : For "alertid" you can create "New Runbook parameter" of String type.

Save the automation step parameter mapping and publish the runbook.

12.5: Update Trigger to set Execution Automatic​

Automations → Runbooks → Triggers -> Edit Trigger "Restart API Service"

Update Runbook execution from "Manual" to "Automatic".

Configure Parameter mapping from alert to Runbook parameters "alertid' = Serial and "Node" = Node

Save changes to the trigger.

12.6: Test Fully automated Runbook​

Connect to OCP Infra node.

Check status of apache2 service on OCP Infra node. If it is not running start it.

sudo systemctl status apache2
sudo systemctl start apache2

Now stop apache2 Service and insert test alert for apache2 Service. (In live monitoring this alert is automatically generated when Service goes down)

sudo systemctl stop apache2
sudo systemctl status apache2

Login into OCP cluster and simulate corresponding alert

oc exec -it evtmanager-ncoprimary-0 -- /bin/bash -c '/opt/IBM/tivoli/netcool/omnibus/bin/nco_sql -server AGG_P -user root -pass $OMNIBUS_ROOT_PWD'
info

Replace '<HOSTNAME>' with the Control Node eth0 IP address captured in 11-5 in below command.

delete from alerts.status where Summary like 'API Server'
go
insert into alerts.status (Identifier,Severity,Type,AlertGroup,Node,FirstOccurrence,LastOccurrence,Manager,Class,Summary) values('Demo API Server <HOSTNAME> 1',5,1,'Demo','<HOSTNAME>',getdate,getdate,'Tivoli EIF Probe',6601,'API Server on <HOST>:<PORT> is DOWN');
go

Expected Output

Launch Alert Viewer and check alert with Summary = " API Server on HOST is DOWN". There should be a runbook associated with this alert automatically.

Open Event information for this alert to which runbook is associated. You will see Runbook parameters under Information, Runbook is set to run automatically.

Once it starts automatic execution "RunbookStatus" changes to "In Progress" -> "Complete"

note

For more information on RunbookStatus : https://www.ibm.com/docs/en/runbook-automation?topic=events-state-transitions-runbookstatus-field

Open Event information for this alert to which runbook is associated. Click the Timeline tab:

Runbook execution status and result are updated for this event.

You can check if the system automatically restarted the service.

Connect to OCP Infra node & check status of apache2 service.

sudo systemctl status apache2

It should return running status , as runbook automatically started this service

Also you can view the runbook execution history in "Automations → Runbooks → Execution" tab.

Now you have a fully automated runbook functional in your environment.

note

For more details on Runbooks, see the documentation link: https://www.ibm.com/docs/en/noi/1.6.5?topic=systems-managing-runbooks-automations

You have now completed this module and learned how to create and configure fully automated runbook in Event Manager.
